# Улучшения buildConnectionString

## 1. Убрать `O.of` обёртку

`O.of(cfg)` + `O.flatMap(({database}) => O.fromNullable(database))` — это лишний шаг. Можно сразу начать с `O.fromNullable(cfg.database)`, без оборачивания всего конфига в Option.

## 2. Разделить валидацию и форматирование

Сейчас в последнем `flatMap` смешаны три вещи: проверка порта через `fromPredicate`, обработка `undefined` порта через `fromNullable` + `fold`, и сборка строки. Это вложенный `pipe` внутри `map` внутри `flatMap` — трудно читать.

Чище: сначала все валидации (host существует, port в диапазоне), потом один финальный `map` для сборки строки.

## 3. `fromPredicate` на объекте, а не на `port`

Применять `fromPredicate` к `port` отдельно — неудобно, потому что `port` может быть `undefined` (валидный случай), и приходится писать `c === undefined ? true : c > 1 && c < 65535`.

Если применить `fromPredicate` ко всему объекту `{ host, port }`, предикат читается естественнее: "пропусти, если порт не задан или в диапазоне".

## 4. Итого

Текущая структура: 3 вложенных `flatMap`, каждый с внутренним `pipe`. Можно свести к:

- `fromNullable` — достать database
- `flatMap` — проверить host, сохранить port рядом
- `flatMap` с `fromPredicate` — валидация порта на уровне объекта
- `map` — одна точка сборки строки
