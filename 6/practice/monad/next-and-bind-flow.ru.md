# Поток `next` и `bind`

Эта заметка объясняет идею `next` в IO DSL.

Главная модель:

```text
IO - это описание программы.
IO само по себе не выполняет эффекты.
```

Каждая инструкция отвечает на один вопрос:

```text
Когда эта инструкция будет интерпретирована, какая IO-программа идет дальше?
```

Эта "следующая IO-программа" и хранится в `next`.

## Форма дерева

IO-программу можно смотреть как на дерево инструкций:

```text
                 IO<A>
                  |
      +-----------+------------+
      |           |            |
   Pure<A>   ReadLine<A>   WriteLine<A>
      |           |            |
   value:A   next: string   text: string
             -> IO<A>       next: IO<A>
```

Важная асимметрия:

```text
ReadLine хранит continuation-функцию.
WriteLine хранит уже готовую следующую программу.
```

## Что такое continuation-функция

`continuation` означает "продолжение программы".

`continuation-функция` - это функция, которая получает результат текущего шага
и строит следующий кусок IO-программы.

Общая форма:

```text
результат текущего шага -> следующая IO-программа
```

Для `readLine` это выглядит так:

```text
string -> IO<A>
```

Смысл:

```text
"Когда появится строка, передай ее в эту функцию.
Функция решит, какая программа должна идти дальше."
```

Почему это важно: следующий шаг часто зависит от результата предыдущего шага.
Если пользователь ввел `"Alice"`, программа может построить одно продолжение.
Если пользователь ввел `"Bob"`, продолжение может быть другим.

Схема:

```text
текущий шаг производит значение
              |
              v
      continuation-функция
              |
              v
      следующая IO-программа
```

Для `writeLine` continuation-функция не нужна, потому что `writeLine` не
производит полезного значения. После записи текста следующая программа уже
известна заранее.

## Pure

`pure` означает, что программа завершена и значение уже есть.

```text
Pure(value)
```

У `pure` нет `next`, потому что больше нет инструкции, которую надо
интерпретировать.

Когда `bind` видит `pure`, он может сразу передать значение в continuation:

```text
bind(Pure(value), f)
```

Схема:

```text
до bind:

  Pure(value)

после bind:

  f(value)
```

Поток:

```text
value уже доступно
=> вызываем f(value)
=> f(value) становится следующей программой
```

## ReadLine

`readLine` производит значение в будущем: строку из внешнего мира.

Поэтому `readLine.next` - это функция:

```text
next: string -> IO<A>
```

В момент построения программы строки еще нет.

Значит, `readLine` не может хранить готовую следующую программу, если эта
программа зависит от пользовательского ввода. Вместо этого он хранит функцию,
которая построит следующую программу после того, как ввод станет известен.

Пример потока:

```text
ReadLine(next)
```

Форма узла:

```text
       ReadLine
          |
          v
 next: input -> IO<A>
```

Поток интерпретатора:

```text
1. Интерпретатор доходит до ReadLine
2. Он просит у мира строку
3. Мир возвращает "Alice"
4. Интерпретатор вызывает next("Alice")
5. next("Alice") возвращает следующую IO-программу
6. Интерпретатор продолжает с этой возвращенной программой
```

Для `readLine` поле `next` означает:

```text
"Дай мне прочитанную строку, и я построю следующую программу."
```

Runtime-схема:

```text
            +-------------+
            |  ReadLine   |
            +------+------+
                   |
                   | интерпретатор просит мир
                   v
              world.readLine()
                   |
                   | возвращает "Alice"
                   v
          вызывается next("Alice")
                   |
                   v
          создается следующая IO-программа
```

## WriteLine

`writeLine` не производит полезного значения.

Он пишет текст и затем продолжает. Нет результата, которого следующий шаг
должен ждать.

Поэтому `writeLine.next` - это уже IO-значение:

```text
next: IO<A>
```

Пример потока:

```text
WriteLine("What is your name?", next)
```

Форма узла:

```text
       WriteLine
       /       \
 text:string   next: IO<A>
```

Поток интерпретатора:

```text
1. Интерпретатор доходит до WriteLine
2. Он отправляет "What is your name?" во внешний мир
3. Полезного возвращаемого значения нет
4. Интерпретатор продолжает с уже сохраненной программой next
```

Для `writeLine` поле `next` означает:

```text
"После записи этого текста продолжай с уже известной программой."
```

Runtime-схема:

```text
          +------------------------------+
          | WriteLine("What is your name?") |
          +---------------+--------------+
                          |
                          | интерпретатор просит мир
                          v
                    world.writeLine(text)
                          |
                          | полезного значения нет
                          v
                    продолжить с next
```

## Почему `readLine.next` - функция, а `writeLine.next` - нет

Разница в том, производит ли инструкция значение, от которого может зависеть
следующий код.

```text
readLine  -> производит string -> next должен получить эту string
writeLine -> производит void   -> next не нужен аргумент
```

Иначе говоря:

```text
readLine.next("Alice") строит следующую программу для Alice
writeLine.next уже является следующей программой
```

## Что пытается сделать `bind`

`bind(io, f)` означает:

```text
Возьми описание программы `io`.
Когда оно когда-нибудь произведет значение, передай это значение в `f`.
Затем продолжи с IO-программой, которую вернула `f`.
```

Но сам `bind` не выполняет эффекты.

Он только переписывает IO-дерево так, чтобы `f` была приклеена к месту, где
появится финальное значение.

Общая схема `bind`:

```text
bind(program, f)

означает:

program когда-нибудь произведет A
          |
          v
       f(A) произведет IO<B>
          |
          v
     весь результат имеет тип IO<B>
```

## Поток через `bind`

### Случай: Pure

```text
bind(Pure(value), f)
```

Значение уже доступно.

Поток:

```text
Pure(value)
=> f(value)
```

Это единственный случай, где `bind` может вызвать `f` сразу.

### Случай: ReadLine

```text
bind(ReadLine(next), f)
```

Строки еще нет. Она появится только тогда, когда интерпретатор запустит
программу.

Поэтому `bind` должен оставить инструкцию `ReadLine`, но заменить ее `next` на
новую функцию.

Концептуальный поток:

```text
старое:
ReadLine(input => oldNext(input))

после bind:
ReadLine(input => bind(oldNext(input), f))
```

Схема переписывания дерева:

```text
до:

        ReadLine
           |
           v
   input -> oldNext(input)

после bind(..., f):

        ReadLine
           |
           v
   input -> bind(oldNext(input), f)
```

Runtime-поток с вводом `"Alice"`:

```text
1. Интерпретатор доходит до ReadLine
2. Мир возвращает "Alice"
3. Новый next получает "Alice"
4. Он вызывает oldNext("Alice")
5. oldNext("Alice") возвращает старую continuation-программу
6. bind приклеивает f глубже в эту возвращенную программу
7. Интерпретатор продолжает выполнение
```

Ключевая мысль:

```text
bind пока не знает ввод.
Он может только подготовить то, что должно случиться после появления ввода.
```

Двухфазная схема:

```text
Фаза 1: время bind, реального ввода еще нет

  ReadLine(next)
       |
       v
  заменить next на wrapper

Фаза 2: runtime, ввод уже есть

  wrapper("Alice")
       |
       v
  oldNext("Alice")
       |
       v
  bind(result, f)
```

### Случай: WriteLine

```text
bind(WriteLine(text, next), f)
```

Сама инструкция записи должна остаться на месте. `bind` не должен ее
пропустить, потому что запись является частью описанной программы.

Но у `writeLine` нет полезного результата, поэтому `f` нельзя приклеить прямо к
самой инструкции записи.

Вместо этого `bind` приклеивает `f` к уже известной программе `next`.

Концептуальный поток:

```text
старое:
WriteLine(text, next)

после bind:
WriteLine(text, bind(next, f))
```

Схема переписывания дерева:

```text
до:

        WriteLine(text)
              |
              v
            next

после bind(..., f):

        WriteLine(text)
              |
              v
        bind(next, f)
```

Runtime-поток:

```text
1. Интерпретатор доходит до WriteLine
2. Мир пишет текст
3. Интерпретатор переходит в новый next
4. Новый next - это старый next, но с f, приклеенной глубже
```

Ключевая мысль:

```text
writeLine.next уже является программой, поэтому bind может сразу рекурсивно идти в нее.
readLine.next является функцией, поэтому bind должен обернуть эту функцию.
```

Сравнение двух случаев:

```text
Случай ReadLine:

  next - это функция

  oldNext: input -> IO<A>
       |
       v
  newNext: input -> bind(oldNext(input), f)


Случай WriteLine:

  next - это уже IO<A>

  oldNext: IO<A>
       |
       v
  newNext: bind(oldNext, f)
```

## Короткая ментальная модель

```text
Pure      -> значение уже здесь, применяем f сейчас
ReadLine  -> значение появится позже, оборачиваем next
WriteLine -> значения здесь нет, рекурсивно идем в next
```

Еще короче:

```text
readLine.next  = "когда придет ввод, построить следующую программу"
writeLine.next = "следующая программа уже известна"
bind           = "приклеить f к месту, где будет произведено значение"
```
