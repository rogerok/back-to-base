## Шаг 1. Типы и интерфейсы

### Уровень 0 — Контракт

Определи базовые типы проекта в отдельных файлах:

- **Methods** — union строковых литералов HTTP-методов (GET, POST, PATCH, DELETE, PUT)
- **Next** — функция без аргументов, возвращающая Promise<void> или void
- **Handler** — функция принимающая контекст и next, возвращающая Promise<void> или void. Handler и Middleware — это один и тот же тип
- **Context** — глобальный интерфейс (через `declare global`), содержащий req и res. Посмотри в тестах на `createReq()` и `createRes()` — они показывают точную форму объектов

> **Именование**: в тестах ментора используется `Context2`, не `Context`. Ты можешь назвать свой интерфейс как хочешь, но при подключении тестов учитывай это различие.

**Тесты**: на этом шаге тесты ещё не запустятся — это фундамент для всего остального.

## Шаг 2. HandlerStorage

### Уровень 0 — Контракт

Класс, который хранит обработчики по HTTP-методам для одного узла дерева.

Методы:
- `addHandler(method, handler)` — сохранить обработчик для метода
- `getHandler(method)` — получить обработчик или null/undefined
- `hasHandlers()` — есть ли хотя бы один обработчик
- `getAllowedMethods()` — вернуть список методов, для которых есть обработчики

Один узел дерева может иметь обработчики для нескольких HTTP-методов одновременно (GET и POST на одном пути — это нормально).

**Тесты**: ещё рано запускать менторские тесты. Можно написать свои unit-тесты.

- Почему `hasHandlers` нужен отдельно от `getHandler`?

## Шаг 3. Node

### Уровень 0 — Контракт

Класс узла radix tree. Каждый узел имеет:

| Поле | Описание |
|------|----------|
| path | Строковый фрагмент пути этого узла (например "users", ":id") |
| type | Тип узла: Static, Parametric или Wildcard |
| children | Массив дочерних узлов |
| paramName | Имя параметра (только для Parametric узлов, иначе null) |
| handlerStorage | Экземпляр HandlerStorage |

Типы узлов можно задать через enum или объект-константу с тремя значениями.

**Тесты**: ещё рано. Это структура данных — проверишь через RouteTree.

<summary><strong>Уровень 1 — Наводящие вопросы</strong></summary>

- Зачем `paramName` отдельным полем, если он уже есть в `path` (после двоеточия)?
- Какие значения по умолчанию для каждого поля при создании узла?

## Шаг 4. RouteTree — insert (без split)

### Уровень 0 — Контракт

Класс с корневым узлом. Метод `insert(method, path, handler)` добавляет маршрут в дерево.

Поведение:
- Путь нормализуется: убирается **только** начальный `/`. Конечный `/` сохраняется! Тесты проверяют что `/items` и `/items/` — разные маршруты
- Если узел с нужным путём уже существует — добавляем обработчик к нему
- Если нет — создаём новые узлы
- Путь может содержать параметры (`:id`) и статические части (`users/`)

Примеры:
- После `insert(GET, "/users", h1)` дерево: root -> "users" с обработчиком GET
- После `insert(GET, "/users/:id", h2)` дерево: root -> "users/" -> :id с обработчиком GET
- После `insert(POST, "/users", h3)` узел "users" получает ещё и POST обработчик

**Тесты**: пока нет — нужен ещё search. Проверяй вручную (отладчик, console.log дерева).

<details>
<summary><strong>Уровень 1 — Наводящие вопросы</strong></summary>

- Когда ты рекурсивно спускаешься по дереву, как определить какой дочерний узел подходит?
- Что значит "подходит"? Полное совпадение path дочернего узла с началом вставляемого пути?
- Когда дочерних узлов нет или ни один не подходит — что делать?
- Если путь содержит `:` не в начале (например `users/:id`) — как разделить на статическую часть и параметр?
- Где именно проходит граница статической части? Включает ли она `/` перед `:`?

</details>

<details>
<summary><strong>Уровень 2 — Алгоритм словами</strong></summary>

**Нормализация**: убери первый символ `/` из пути. Больше ничего не трогай.

**Рекурсивная вставка** принимает текущий узел и оставшийся путь:

1. Если путь пустой — мы на месте. Добавь обработчик в текущий узел.

2. Переберём статических потомков текущего узла. Для каждого найди общий префикс между path потомка и оставшимся путём. Если prefix полностью совпадает с path потомка — продолжи рекурсию в этого потомка, отрезав совпавшую часть от пути. (Случай частичного совпадения — это split, пока пропусти.)

3. Если путь начинается с `:` — проверь, есть ли среди потомков параметрический узел с таким же именем параметра. Если есть — продолжи рекурсию с остатком пути (после параметра до следующего `/` или конца строки). Если нет — создай новый параметрический узел.

4. Если ни один потомок не подошёл — создай новый узел. Тут важный момент: если путь содержит `:` или `*` НЕ в начале (например `users/:id`), нужно разбить его на статическую часть до спецсимвола и остаток. Статическая часть включает `/` перед `:`. Создай статический узел для первой части, а для остатка — продолжи рекурсию.

</details>

## Шаг 5. RouteTree — insert (со split)

### Уровень 0 — Контракт

Добавь обработку ситуации, когда общий префикс вставляемого пути и path существующего узла **короче**, чем path узла.

Пример ситуации:
- В дереве: root -> "testing" с обработчиком GET
- Вставляем: "test" с обработчиком GET
- Общий префикс: "test" (4 символа), а path узла: "testing" (7 символов)
- Нужно разделить узел "testing" на "test" и "ing"

После split:
- root -> "test" (новый обработчик GET) -> "ing" (старый обработчик GET)

**Тесты**: пока нет — нужен ещё search.

<details>
<summary><strong>Уровень 1 — Наводящие вопросы</strong></summary>

- Когда ты разделяешь узел, что происходит с его обработчиками? Кому они достаются — "верхнему" или "нижнему" узлу?
- А что с потомками разделяемого узла?
- После split, куда вставляется новый маршрут? В верхний узел или нужна ещё одна итерация?
- Как определить позицию разреза? Это длина общего префикса.

</details>

<details>
<summary><strong>Уровень 2 — Алгоритм словами</strong></summary>

**split(node, позиция)** — разделяет узел на два:

1. Отрезаем "хвост" — часть path узла после позиции. Создаём новый дочерний узел с этим хвостом.
2. Переносим в этот новый узел все обработчики и всех потомков исходного узла.
3. Исходный узел укорачиваем: его path = только первая часть (до позиции). Очищаем его обработчики. Его единственный потомок — новый узел из пункта 1.

После split вызови рекурсивную вставку для оставшейся части пути (за пределами общего префикса) — она создаст второго потомка или добавит обработчик в сам укороченный узел.

</details>

## Шаг 6. RouteTree — search

### Уровень 0 — Контракт

Метод `search(method, path)` ищет маршрут в дереве.

Возвращает объект с обработчиком и извлечёнными параметрами, либо null.

Поведение:
- Нормализация пути — аналогично insert (только leading slash)
- Приоритет поиска среди потомков: **Static > Parametric > Wildcard**
- Статический маршрут `/users/me` побеждает параметрический `/users/:id`
- Параметрические сегменты извлекают значения: путь `/users/42` с маршрутом `/users/:id` даёт params = { id: "42" }
- Если ветка зашла в тупик — нужен откат (backtracking) к следующему варианту

**Тесты**: после этого шага напиши свои unit-тесты для RouteTree — вставляй маршруты и проверяй поиск.

<details>
<summary><strong>Уровень 1 — Наводящие вопросы</strong></summary>

- Как определить, что статический потомок "подходит"? Что значит `path начинается с child.path`?
- Для параметрического потомка — где заканчивается значение параметра? Какой символ служит разделителем?
- Что если параметрическая ветка зашла в тупик (рекурсия вернула null)? Нужно откатить добавленный параметр и попробовать следующего потомка.
- Почему важен порядок обхода потомков? Что случится если Parametric проверять раньше Static?
- Когда рекурсия дошла до пустого пути — как проверить что это валидный конечный узел, а не промежуточный?

</details>

<details>
<summary><strong>Уровень 2 — Алгоритм словами</strong></summary>

**search** нормализует путь, создаёт пустой объект params, запускает рекурсивный поиск от корня. Если рекурсия вернула узел — проверяет есть ли у него обработчик для нужного метода.

**Рекурсивный поиск** принимает узел, оставшийся путь и объект params:

1. Путь пустой — мы на месте. Возвращай текущий узел если у него есть обработчики, иначе null.

2. Перебирай статических потомков. Если оставшийся путь начинается с path потомка — рекурсия вглубь с укороченным путём. Если рекурсия вернула результат — возвращай его.

3. Перебирай параметрических потомков. Значение параметра — это часть пути от начала до первого `/` (или весь путь если `/` нет). Запиши значение в params под именем параметра. Рекурсия вглубь с остатком пути. Если рекурсия вернула null — **удали** параметр из params (откат) и пробуй следующего потомка.

4. Перебирай wildcard-потомков. Wildcard захватывает весь оставшийся путь целиком.

5. Ничего не нашли — возвращай null.

</details>

## Шаг 7. compose()

### Уровень 0 — Контракт

Функция `compose` принимает массив middleware и возвращает одну функцию с такой же сигнатурой (ctx, next) => Promise<void>.

Поведение:
- Middleware вызываются по порядку (первая в массиве — первая выполняется)
- Каждая middleware получает `next`, вызов которого запускает следующую middleware
- `await next()` ждёт завершения всех последующих middleware (onion model)
- Если middleware не вызывает next — цепочка прерывается
- Если middleware бросает ошибку — Promise реджектится
- Вызов next() дважды в одной middleware — ошибка
- Синхронные и асинхронные middleware работают одинаково

Проверь на примере: [mw1-in, mw2-in, handler, mw2-out, mw1-out] — это "луковица".

**Тесты**: `"middleware"`, `"routes as middleware"` — проверяют onion-порядок, short-circuit, ошибки. Но для запуска нужен ещё Router.

> Эталонная реализация — [koa-compose](https://github.com/koajs/compose/blob/master/index.js), ~25 строк.

<details>
<summary><strong>Уровень 1 — Наводящие вопросы</strong></summary>

- Как передать управление от middleware[i] к middleware[i+1]? Что такое `next` для i-й middleware?
- Как обеспечить что `await next()` действительно ждёт? Подсказка: Promise.
- Как защититься от двойного вызова next()? Какое состояние нужно хранить?
- Что происходит когда все middleware отработали? Что вызывается последним?
- Как обернуть синхронную функцию чтобы она тоже возвращала Promise?

</details>

<details>
<summary><strong>Уровень 2 — Алгоритм словами</strong></summary>

Возвращаем функцию, которая запускает диспетчер с индексом 0.

**Диспетчер(i)** — рекурсивная/итеративная функция:

1. Запомни текущий индекс для защиты от двойного вызова. Если диспетчер уже вызывался с таким или меньшим индексом — это ошибка.
2. Возьми middleware по индексу i. Если индекс вышел за массив — возьми финальный next (переданный снаружи). Если и его нет — просто разрешись.
3. Вызови middleware, передав ей ctx и функцию-обёртку, которая вызовет диспетчер(i+1).
4. Оберни результат в Promise.resolve() чтобы поддержать синхронные middleware.
5. Лови синхронные исключения и превращай в Promise.reject().

</details>

## Шаг 8. Router — каркас и handle()

### Уровень 0 — Контракт

Класс Router с полями:
- Экземпляр RouteTree
- Массив текущих middleware (изначально пустой)

Метод `handle(req, res)` — точка входа для обработки запроса. Возвращает Promise<void>.

Поведение handle:
- Отделяет path от query string в URL
- Парсит query string в объект ключ-значение
- Ищет маршрут в дереве по методу и path
- Если маршрут не найден — возвращается без действий (не бросает ошибку)
- Если найден — создаёт **новый** объект контекста с req (включая params и query) и res
- Вызывает найденный обработчик с контекстом

Все методы (`use`, `get`, `post`, `patch`, `delete`, `nest`) возвращают `this` для chaining.

**Тесты**: `"chainable API"`, `"handler context"`, `"edge cases"` → "no registered routes".

<details>
<summary><strong>Уровень 1 — Наводящие вопросы</strong></summary>

- Как отделить path от query string? Какой символ их разделяет?
- Есть ли стандартный API в JavaScript для парсинга query string?
- Почему контекст должен быть **новым** для каждого запроса? Что произойдёт если его переиспользовать?
- Обработчик в дереве — это уже скомпонованная функция (compose вызвался при регистрации). Что handle() передаёт вторым аргументом при вызове?

</details>

<details>
<summary><strong>Уровень 2 — Алгоритм словами</strong></summary>

**handle(req, res)**:

1. Найди позицию `?` в URL. Всё до неё — path, всё после — query string. Если `?` нет — path это весь URL, query пустой.
2. Для парсинга query string используй URLSearchParams — он умеет разбирать строку вида `key=value&key2=value2` в пары. Преврати результат в обычный объект.
3. Вызови search дерева с методом запроса и path.
4. Если результат null — return.
5. Создай объект контекста: req содержит все поля из оригинального req плюс params из результата поиска и распарсенный query. res передаётся как есть.
6. Вызови обработчик из результата поиска, передав контекст и пустую async-функцию (как финальный next).

</details>

## Шаг 9. HTTP-методы и middleware

### Уровень 0 — Контракт

**Методы .get(), .post(), .patch(), .delete()** — каждый принимает path и handler, регистрирует маршрут в дереве. Все возвращают `this`.

**Метод .use()** — принимает middleware, добавляет в текущий стек. Возвращает `this`.

Ключевое поведение (snapshot semantics):
- При регистрации маршрута (.get и т.д.) делается **снимок** текущего массива middleware
- Middleware добавленная **после** регистрации маршрута **не влияет** на него
- Пример: `.get("/a", h1).use(mw).get("/b", h2)` — у маршрута "/a" нет middleware, у "/b" есть mw

**Тесты**: `"chainable API"`, `"HTTP method routing"`, `"middleware"` (все), `"routes as middleware"` (все).

<details>
<summary><strong>Уровень 1 — Наводящие вопросы</strong></summary>

- Как сделать snapshot массива чтобы дальнейшие push не влияли на него?
- Если handler — это "последняя middleware в цепочке", что получится если скомпоновать snapshot middleware и handler в одну функцию через compose? Что тогда хранится в дереве?
- Зачем compose'ить при регистрации, а не при каждом запросе? Подумай о производительности и о snapshot semantics.
- Как DRY-но реализовать четыре метода (get/post/patch/delete), различающихся только строкой метода?

</details>

<details>
<summary><strong>Уровень 2 — Алгоритм словами</strong></summary>

**use(middleware)**: просто добавь middleware в массив currentMiddleware. Верни this.

**Приватный метод addRoute(method, path, handler)**:
1. Скопируй текущий массив middleware (snapshot).
2. Собери массив: все middleware из snapshot, затем handler в конце.
3. Скомпонуй этот массив через compose — получишь одну функцию.
4. Вставь скомпонованную функцию в дерево под нужным методом и путём.

**get(path, handler)**: вызови addRoute("GET", path, handler), верни this. Аналогично для post, patch, delete.

Благодаря тому что в дерево попадает уже скомпонованная функция, handle() не знает ни про какие middleware — он просто вызывает одну функцию.

</details>

## Шаг 10. Nesting

### Уровень 0 — Контракт

Метод `nest(prefix, callback)` — создаёт область видимости с префиксом пути. Возвращает `this`.

Поведение:
- callback получает дочерний роутер, в который можно добавлять маршруты и middleware
- Маршруты дочернего роутера получают prefix: `nest("/api", r => r.get("/users", h))` регистрирует маршрут `/api/users`
- Parent middleware наследуется дочерним роутером
- Middleware добавленная внутри nest НЕ утекает наружу
- Поддерживается глубокая вложенность: nest внутри nest

**Тесты**: `"nesting"` (все 7 тестов), `"radix tree routing"` → "parametric route in nested scope", "parametric route at root of nested scope".

<details>
<summary><strong>Уровень 1 — Наводящие вопросы</strong></summary>

- Если дочерний роутер — это новый экземпляр Router, что нужно в него "передать" от родителя чтобы parent middleware работала?
- После того как callback отработал, маршруты живут в дереве дочернего роутера. Как их "перенести" в дерево родителя? Какая информация нужна для каждого маршрута?
- Radix tree хранит пути в сжатом виде (разбитые по узлам). Чтобы "собрать" полные пути маршрутов — нужно обойти дерево от корня до каждого узла с обработчиками. Как это сделать рекурсивно?
- При сборке полного пути из фрагментов узлов — как обработать параметрические узлы? Перед `:id` нужен `/`, но статический узел может уже заканчиваться на `/`.
- Альтернативный подход: вместо обхода дерева, можно ли перехватить регистрацию маршрутов в дочернем роутере и сразу вставлять их с prefix в родительское дерево?

</details>

<details>
<summary><strong>Уровень 2 — Алгоритм словами</strong></summary>

**nest(prefix, callback)**:
1. Создай новый Router (дочерний).
2. Скопируй текущий массив middleware родителя в дочерний (snapshot).
3. Вызови callback(child) — пользователь наполнит дочерний роутер маршрутами.
4. После callback — "собери" все маршруты из дерева дочернего роутера и вставь их в дерево родителя с добавлением prefix.

**Сбор маршрутов (collectRoutes)** — рекурсивный обход дерева:

Принимает узел и путь, накопленный от корня до этого узла.

1. Если текущий узел имеет непустой path — добавь его к накопленному пути.
2. Если у узла есть обработчики — запомни полный путь (с leading `/` если его нет) и обработчики.
3. Для каждого дочернего узла определи prefix для рекурсии: для параметрических и wildcard потомков между накопленным путём и потомком нужен `/` как разделитель. Но если накопленный путь уже заканчивается на `/` — не дублируй его.
4. Рекурсия в потомков.

После сбора: для каждого маршрута вставь его обработчики в родительское дерево с путём = prefix + собранный путь.

</details>

## Шаг 11. Zod-валидация

### Уровень 0 — Контракт

Расширяем HTTP-методы: помимо `get(path, handler)` добавляем перегрузку `get(path, options, handler)`, где options содержит объект schema.

Schema может валидировать:
- **body** — тело запроса (невалидное → ответ 400, handler не вызывается)
- **params** — параметры маршрута (невалидные → 400)
- **query** — параметры запроса (невалидные → 400)
- **response** — тело ответа по статус-коду (невалидное → throw Error)

Для проверки body, params, query используй `safeParse()` из Zod — он не бросает исключение, а возвращает объект с полем `success`.

Валидация response отличается: нужно перехватить вызов `res.status(code).send(data)` и провалидировать data перед отправкой.

**Тесты**: `"schema validation"` (все 8 тестов).

<details>
<summary><strong>Уровень 1 — Наводящие вопросы</strong></summary>

- Как в TypeScript реализовать две разные сигнатуры для одной функции? Как внутри реализации определить какой вариант вызвали?
- Валидация body/params/query — это по сути middleware, которая проверяет данные перед handler'ом. Куда в массив middleware её поставить?
- Для response validation — как "перехватить" вызов send()? Можно ли заменить метод объекта на обёртку?
- Почему response validation тоже middleware, а не пост-обработка? Подсказка: onion model — middleware может выполнять код и до, и после next().
- Зачем response validation бросает исключение, а не отвечает 400? Чем валидация ответа отличается от валидации запроса?

</details>

<details>
<summary><strong>Уровень 2 — Алгоритм словами</strong></summary>

**Function overloading**: определи две сигнатуры (path+handler и path+opts+handler), а в реализации проверяй тип второго аргумента: если это функция — это handler, если объект — это opts (а handler будет третьим аргументом).

**Validation middleware для body/params/query**: создай функцию, которая возвращает middleware. Эта middleware перед вызовом next() проверяет каждую указанную схему через safeParse. Если хотя бы одна проверка не прошла — отвечает status(400).send({error: ...}) и НЕ вызывает next(). Если все прошли — вызывает next().

**Response validation middleware**: создай middleware, которая перед вызовом next() подменяет метод status у res на обёртку. Обёртка вызывает оригинальный status, но подменяет send у возвращённого объекта. Подменённый send проверяет данные через safeParse по соответствующей схеме для этого статус-кода. Если проверка не прошла — бросает исключение. Если прошла — вызывает оригинальный send. После подмены вызывается await next() — handler вызовет перехваченные методы.

**Сборка**: при регистрации маршрута с opts добавь validation middleware в массив перед handler'ом (после snapshot общих middleware). Если есть response schema — добавь response validation middleware тоже.

</details>

## Шаг 12. Типизированные ответы

### Уровень 0 — Контракт

Определи типы, которые связывают статус-код ответа с типом тела:

- **StatusBodyMap** — интерфейс-маппинг, где ключ — числовой код (200, 201, 400...), значение — тип тела ответа для этого кода
- **StatusCode** — union числовых литералов из ключей StatusBodyMap
- Типизированный метод `status(code)` возвращает объект, где `send` принимает тип, соответствующий этому коду

Это работает только на уровне типов — runtime-поведение не меняется.

**Тесты**: типы проверяются косвенно через TypeScript компиляцию, а не через runtime-тесты.

<details>
<summary><strong>Уровень 1 — Наводящие вопросы</strong></summary>

- Как TypeScript понимает что `status(200).send(...)` требует один тип, а `status(400).send(...)` — другой? Подсказка: generic параметр + indexed access type.
- Что означает запись `StatusBodyMap[S]` где S extends StatusCode?
- Как сделать StatusCode автоматически выводимым из ключей StatusBodyMap, а не дублировать?

</details>

<details>
<summary><strong>Уровень 2 — Алгоритм словами</strong></summary>

Определи интерфейс StatusBodyMap с числовыми ключами (200, 201, 204, 400, 401, 404, 500) и типами тел для каждого (объект с данными, never для 204, объект с полем error для ошибок).

StatusCode — это `keyof StatusBodyMap` (TypeScript сам выведет union числовых литералов).

В интерфейсе res: метод `status` принимает generic параметр S extends StatusCode и возвращает объект, у которого метод `send` принимает `StatusBodyMap[S]`. Благодаря этому TypeScript при вызове `status(400)` подставит S=400 и send потребует `StatusBodyMap[400]`.

</details>
