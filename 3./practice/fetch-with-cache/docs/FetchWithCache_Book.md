# Сетевые запросы, отказоустойчивость и кэширование

Учебное пособие для самостоятельной реализации `fetchWithCache`.

---

## Глава 1. Сеть ненадёжна

### 1.1. Заблуждение новичка

Когда ты пишешь `fetch("https://api.example.com/data")`, ты подсознательно
представляешь это как вызов функции — позвал, получил результат. Как будто
данные лежат где-то рядом, в соседней переменной.

На самом деле между твоим `fetch` и ответом сервера происходит вот что:

```
Твой код
  → браузер формирует HTTP-запрос
    → DNS-резолвер ищет IP-адрес сервера
      → пакет уходит через твой роутер
        → через провайдера
          → через несколько маршрутизаторов интернета
            → в дата-центр сервера
              → на балансировщик нагрузки
                → на один из десятков серверов
                  → сервер обрабатывает запрос
                → ответ идёт обратно
              → через балансировщик
            → через маршрутизаторы
          → через провайдера
        → через роутер
      → в браузер
    → браузер парсит ответ
  → твой код получает Response
```

На любом из этих шагов что-то может пойти не так. Провод обрезали,
сервер перегружен, DNS лежит, роутер перезагрузился. Сеть — это не
функция. Это цепочка из десятков звеньев, каждое из которых может
сломаться.

Питер Дейч в 1994 году сформулировал
[8 заблуждений о распределённых системах](https://en.wikipedia.org/wiki/Fallacies_of_distributed_computing).
Первое заблуждение: **"Сеть надёжна"**. Это не так, и твой код должен
это учитывать.

### 1.2. Две категории ошибок

Когда запрос не удался, важно понять **почему**. От этого зависит,
что делать дальше.

**Транзиентные (временные) ошибки — можно повторить:**
- Сервер перегружен (503 Service Unavailable)
- Сервер сломался, но перезапустится (500 Internal Server Error)
- Прокси не достучался до бэкенда (502 Bad Gateway)
- Прокси не дождался ответа (504 Gateway Timeout)
- Сетевой сбой (обрыв соединения)

Через секунду, десять секунд, минуту — проблема может решиться сама.
Повторный запрос имеет смысл.

**Перманентные (постоянные) ошибки — повторять бессмысленно:**
- Ресурс не существует (404 Not Found)
- Нет авторизации (401 Unauthorized)
- Нет прав доступа (403 Forbidden)
- Невалидный запрос (400 Bad Request)

Сколько ни повторяй — результат будет тот же. Если ресурса нет на
сервере, он не появится через секунду от того, что ты спросишь ещё раз.

**Как отличить по HTTP-статусу:**
Статусы группируются в "сотни":
- `2xx` — успех (200 OK, 201 Created, 204 No Content)
- `4xx` — ошибка клиента (перманентная)
- `5xx` — ошибка сервера (транзиентная)

Это не абсолютное правило (429 Too Many Requests — транзиентная, хотя 4xx),
но для твоей задачи деления на 4xx/5xx достаточно.

### 1.3. Что значит "обработать ошибку"

У тебя есть три стратегии:

1. **Сдаться** — вернуть ошибку вызывающему коду. "Не получилось, разбирайся сам."
2. **Повторить** — попробовать ещё раз. Но сколько раз? С какой задержкой?
3. **Обойти** — использовать альтернативный источник данных (кэш, fallback).

Хорошая система комбинирует все три: **сначала проверить кэш** (обойти),
если нет — **запросить и при ошибке повторить**, а если всё равно не
получилось — **сдаться**.

## Глава 2. Кэширование — память системы

### 2.1. Зачем кэшировать

Допустим, ты запросил `https://api.example.com/user/42` и получил
`{ name: "Alice", age: 30 }`. Через секунду тебе нужны те же данные.
Зачем снова идти по всей цепочке из Главы 1.1, если ответ уже есть?

Кэш — это **локальная копия** данных, которые дорого получать.
"Дорого" может означать:
- Долго (сетевой запрос: 50-500ms vs чтение из памяти: <1ms)
- Ресурсоёмко (нагрузка на сервер)
- Ненадёжно (сеть может быть недоступна)

### 2.2. Компромисс кэширования

Кэш — это всегда компромисс между **свежестью** и **скоростью**.

```
Без кэша:     всегда свежие данные, но каждый запрос идёт в сеть
С кэшем:      быстрый доступ, но данные могут быть устаревшими
```

Для твоей задачи это не проблема — ты кэшируешь в рамках одного
"пакета" запросов. Но в реальных приложениях это ключевой вопрос:
как долго хранить кэш? Когда инвалидировать?

> "There are only two hard things in Computer Science:
> cache invalidation and naming things."
> — Phil Karlton

### 2.3. LRU — стратегия вытеснения

Кэш ограничен по размеру (память не бесконечна). Когда он заполнен
и приходит новый элемент — нужно решить, что выбросить.

**LRU (Least Recently Used)** — выбрасываем то, к чему **дольше всего
не обращались**.

Интуиция: если к данным давно не обращались — скорее всего, они не нужны.
А те, к которым обращались недавно — вероятно, понадобятся снова.

Это работает благодаря **принципу локальности** — в большинстве программ
доступ к данным не равномерный. Есть "горячие" данные (часто используемые)
и "холодные" (редко). LRU автоматически держит горячие и вытесняет холодные.

**У тебя уже есть реализация LRU** в
`2.oop-structures/practice/memoize/lru-cache.ts`. Она использует `Map`,
который в JavaScript сохраняет порядок вставки — это свойство позволяет
реализовать LRU за O(1) без связного списка.

Подробнее о стратегиях вытеснения:
[Wikipedia — Cache replacement policies](https://en.wikipedia.org/wiki/Cache_replacement_policies)

## Глава 3. Параллельность в JavaScript

### 3.1. Один поток, много задач

JavaScript однопоточный. Но сетевые запросы — это I/O операции, которые
выполняются **вне** основного потока (в сетевом стеке браузера/ОС).
JavaScript только **инициирует** запрос и **обрабатывает** результат.
Само ожидание ответа не блокирует поток.

Это значит, что ты можешь запустить 10 fetch одновременно:

```ts
// Все три запроса уходят в сеть СРАЗУ, не дожидаясь друг друга
const p1 = fetch(url1);  // → запрос ушёл
const p2 = fetch(url2);  // → запрос ушёл
const p3 = fetch(url3);  // → запрос ушёл

// Теперь ждём все три
```

Это не "настоящая" параллельность (нет потоков), это **конкурентность** —
несколько операций "в полёте" одновременно. Но для сетевых запросов
эффект тот же: вместо 3 последовательных запросов по 200ms (= 600ms)
ты получаешь 3 параллельных за ~200ms.

### 3.2. Как собрать результаты параллельных операций

У тебя есть массив промисов. Нужно дождаться всех. В JavaScript есть
четыре инструмента для этого:

**`Promise.all(promises)`**
- Ждёт, пока все fulfilled
- Если **хоть один** rejected — весь `Promise.all` rejected
- Результаты остальных промисов **теряются**

**`Promise.allSettled(promises)`**
- Ждёт **все** промисы до конца
- Никогда не rejected сам
- Возвращает массив `{ status, value/reason }` для каждого промиса

**`Promise.race(promises)`**
- Возвращает результат **первого** завершившегося (fulfilled или rejected)

**`Promise.any(promises)`**
- Возвращает результат **первого fulfilled**
- Rejected только если все rejected

Для твоей задачи подумай: если один URL вернул ошибку, хочешь ли ты
потерять результаты остальных? Или хочешь получить полный отчёт —
что удалось, что нет?

Подробное сравнение:
[javascript.info — Promise API](https://javascript.info/promise-api)

### 3.3. Паттерн: параллельный запуск с общим контекстом

Когда несколько параллельных операций должны **влиять друг на друга**
(одна ошибка отменяет остальные), нужен **общий контекст** — объект,
доступный всем операциям:

```ts
// Общий контекст — все операции видят один и тот же объект
const sharedContext = { shouldStop: false };

const promises = urls.map(async (url) => {
  if (sharedContext.shouldStop) return;

  const result = await doSomething(url);

  if (isCriticalError(result)) {
    sharedContext.shouldStop = true;  // сигнал остальным
  }

  return result;
});
```

Это грубый пример. В реальности вместо `sharedContext` используется
`AbortController` — стандартный механизм для этого.

## Глава 4. Отмена операций — AbortController

### 4.1. Проблема: промис нельзя отменить

Промис в JavaScript — это **необратимое обязательство**. Когда ты создал
промис, он будет либо fulfilled, либо rejected. Третьего не дано.
У промиса нет метода `cancel()`.

```ts
const promise = fetch(url);
// Как отменить? promise.cancel()? Нет такого метода.
```

Это дизайн-решение. Промисы спроектированы как простой примитив.
Но для сетевых запросов отмена необходима: пользователь ушёл со страницы,
таймаут истёк, результат больше не нужен.

### 4.2. Решение: сигнал отмены

Вместо того чтобы отменять промис, отменяют **операцию внутри промиса**.

Аналогия из жизни: ты заказал пиццу по телефону (fetch). Заказ уже
в процессе (промис pending). Ты не можешь "отменить" сам факт звонка,
но можешь **позвонить ещё раз и сказать "отмените заказ"**.

`AbortController` — это механизм "второго звонка":

```ts
const controller = new AbortController();
//    ^^^^^^^^^^
//    "телефон для отмены"

fetch(url, { signal: controller.signal });
//                   ^^^^^^^^^^^^^^^^^
//                   "привязка заказа к телефону отмены"

controller.abort();
//         ^^^^^^^
//         "звонок: отмените заказ"
```

### 4.3. Как fetch реагирует на abort

Внутри fetch есть примерно такая логика:

```
1. Получить signal из options
2. Если signal.aborted === true → сразу бросить AbortError
3. Начать HTTP-запрос
4. Пока ждём ответ — слушать signal
5. Если signal получил "abort" → прервать соединение, бросить AbortError
6. Если ответ пришёл раньше → вернуть Response
```

`AbortError` — это обычная ошибка, которую можно поймать через `catch`.
Но у неё специальное имя (`err.name === "AbortError"`), по которому
её можно отличить от сетевых и других ошибок.

### 4.4. Один signal — много потребителей

Signal — это **broadcast** механизм. Один signal можно передать в
любое количество fetch-запросов. Когда вызывается `abort()` — **все**
потребители этого signal получают сигнал отмены одновременно.

Это именно то, что нужно для "отменить все запросы":
```ts
const controller = new AbortController();

// 10 запросов — один signal
urls.map(url => fetch(url, { signal: controller.signal }));

// Одним действием отменяем все 10
controller.abort();
```

### 4.5. Signal можно слушать самому

`AbortSignal` — это `EventTarget`. Можно подписаться на событие `"abort"`:

```ts
signal.addEventListener("abort", () => {
  console.log("Отмена!");
});
```

Это полезно, когда ты хочешь сделать свою операцию отменяемой.
Например, `setTimeout` не поддерживает signal из коробки, но ты
можешь научить его:

```ts
function cancellableSleep(ms, signal) {
  return new Promise((resolve, reject) => {
    const timeoutId = setTimeout(resolve, ms);

    signal.addEventListener("abort", () => {
      clearTimeout(timeoutId);
      reject(new DOMException("Aborted", "AbortError"));
    }, { once: true });
  });
}
```

Подумай: зачем это нужно в контексте retry? Что произойдёт, если
между попытками retry (во время паузы) придёт сигнал отмены?

MDN — AbortController:
[developer.mozilla.org/en-US/docs/Web/API/AbortController](https://developer.mozilla.org/en-US/docs/Web/API/AbortController)

javascript.info — Fetch: Abort:
[javascript.info/fetch-abort](https://javascript.info/fetch-abort)

## Глава 5. Retry — искусство повторных попыток

### 5.1. Наивный retry и его проблемы

Простейший retry — повторить через фиксированное время:

```
Ошибка → подождать 1 сек → повторить → ошибка → подождать 1 сек → повторить
```

Это работает для одного клиента. Но представь:

Сервер обслуживает 10 000 запросов в секунду. Его лимит — 12 000.
Произошёл кратковременный сбой на 2 секунды. За это время накопилось
20 000 неудачных запросов.

Если все 20 000 клиентов повторят через 1 секунду:
```
Секунда 0:  сбой
Секунда 1:  20 000 retry → сервер снова перегружен → сбой
Секунда 2:  20 000 retry → сервер снова перегружен → сбой
Секунда 3:  20 000 retry → ...
```

Сервер **никогда** не восстановится. Retry-запросы сами стали причиной сбоя.
Это называется **retry storm** (шторм повторных запросов).

### 5.2. Exponential Backoff — экспоненциальная задержка

Идея: каждая следующая попытка ждёт **вдвое дольше** предыдущей.

```
Попытка 1: подождать 100ms
Попытка 2: подождать 200ms
Попытка 3: подождать 400ms
Попытка 4: подождать 800ms
Попытка 5: подождать 1600ms
```

Формула: `delay = baseDelay * 2^attempt`

Почему это помогает:
- На первой попытке нагрузка на сервер = N запросов
- На второй (через 200ms) часть клиентов уже получила ответ → нагрузка < N
- На третьей (через 400ms) ещё меньше клиентов осталось
- Нагрузка **экспоненциально убывает** с каждой попыткой

Сервер получает время на восстановление, потому что каждая волна retry
меньше предыдущей.

### 5.3. Thundering Herd — проблема синхронности

Exponential backoff решает проблему нарастающей нагрузки, но не решает
проблему **одновременности**.

Если 10 000 клиентов получили ошибку **в одну и ту же миллисекунду**,
они все вычислят одинаковую задержку:

```
Клиент 1:     delay = 100 * 2^0 = 100ms
Клиент 2:     delay = 100 * 2^0 = 100ms
Клиент 3:     delay = 100 * 2^0 = 100ms
...
Клиент 10000: delay = 100 * 2^0 = 100ms
```

Через 100ms — залп из 10 000 запросов. Это **thundering herd**
("стадо бизонов") — все бегут в одну дверь одновременно.

Название пришло из Unix-мира: когда процесс освобождает ресурс и
будит **все** ожидающие процессы разом, они все бросаются к ресурсу
одновременно, создавая пик нагрузки.

### 5.4. Jitter — рандомизация задержки

Решение thundering herd — добавить **случайность**:

```
Клиент 1:     delay = random(0, 100ms)  → 37ms
Клиент 2:     delay = random(0, 100ms)  → 82ms
Клиент 3:     delay = random(0, 100ms)  → 5ms
Клиент 4:     delay = random(0, 100ms)  → 91ms
```

Retry-запросы **размазываются** по временному окну вместо того, чтобы
прийти одновременно. Сервер получает их постепенно и может обработать.

Есть несколько вариантов jitter:

**Full Jitter** (рекомендует AWS):
```
delay = random(0, baseDelay * 2^attempt)
```
Максимально агрессивная рандомизация. Delay может быть от 0 до
максимума. В среднем нагрузка распределяется наиболее равномерно.

**Equal Jitter:**
```
half = baseDelay * 2^attempt / 2
delay = half + random(0, half)
```
Гарантирует минимальную задержку (половину от backoff), но добавляет
случайность во вторую половину.

**Decorrelated Jitter:**
```
delay = random(baseDelay, previousDelay * 3)
```
Каждая следующая задержка зависит от предыдущей.

Для твоей задачи Full Jitter — самый простой и эффективный вариант.

Оригинальная статья AWS с графиками и сравнениями:
[aws.amazon.com/blogs/architecture/exponential-backoff-and-jitter](https://aws.amazon.com/blogs/architecture/exponential-backoff-and-jitter/)

### 5.5. Cap — потолок задержки

Без ограничения exponential backoff растёт бесконечно:

```
attempt 0:   100ms
attempt 5:   3 200ms
attempt 10:  102 400ms  (1.7 минуты!)
attempt 15:  3 276 800ms  (55 минут!)
```

Нужен потолок (cap) — максимальная задержка, после которой backoff
перестаёт расти:

```
delay = min(cap, baseDelay * 2^attempt)
```

С cap = 5000ms:
```
attempt 0:   100ms
attempt 5:   3 200ms
attempt 6:   5 000ms  (cap)
attempt 7:   5 000ms  (cap)
```

Типичные значения cap: 5-30 секунд.

### 5.6. Максимальное количество попыток

Retry не должен быть бесконечным. Если после N попыток сервер так
и не ответил — нужно сдаться и вернуть ошибку.

Типичное значение: 3-5 попыток. Больше обычно не имеет смысла —
если сервер не ответил после 5 попыток с нарастающими задержками,
проблема серьёзнее, чем кратковременный сбой.

### 5.7. Что retry-ить, а что нет — сводная таблица

```
Ситуация                    │ Retry? │ Почему
────────────────────────────┼────────┼────────────────────────────
200 OK                      │ Нет    │ Успех, retry не нужен
4xx (400, 401, 403, 404)    │ Нет    │ Ошибка клиента, повтор не поможет
5xx (500, 502, 503, 504)    │ Да     │ Ошибка сервера, может пройти
Сетевая ошибка (TypeError)  │ Да     │ Временный сбой сети
AbortError                  │ Нет    │ Намеренная отмена, не ошибка
```

**Важно:** AbortError — это **не ошибка**. Это сигнал, что кто-то
намеренно отменил операцию. Retry-ить AbortError — это как пытаться
заново набрать номер после того, как ты сам повесил трубку.

## Глава 6. Fetch API — тонкости, которые кусают

### 6.1. fetch не считает 4xx/5xx ошибкой

Это самый частый источник багов при работе с fetch. Сравни с другими
HTTP-библиотеками:

```ts
// axios — бросает ошибку при 4xx/5xx
try {
  const response = await axios.get(url);
} catch (err) {
  // Сюда попадёт и 404, и 500, и сетевая ошибка
}

// fetch — НЕ бросает ошибку при 4xx/5xx
try {
  const response = await fetch(url);
  // response.status может быть 404 или 500,
  // но мы в try-блоке, а не в catch!
} catch (err) {
  // Сюда попадёт ТОЛЬКО сетевая ошибка или AbortError
}
```

Философия fetch: "Я сделал запрос и получил ответ. Какой бы ни был
статус — это ответ сервера, а не ошибка."

Для fetch ошибка — это когда ответа **вообще нет**: сеть упала,
DNS не резолвится, запрос отменён.

**Что это значит для тебя:** ты должен **сам** проверять статус
и решать, что делать. `response.ok` — shortcut для
`response.status >= 200 && response.status < 300`.

### 6.2. Response.body — одноразовый поток

```ts
const response = await fetch(url);
const data1 = await response.json();   // ✓ работает
const data2 = await response.json();   // ✗ TypeError: body already consumed
```

Тело ответа — это **поток** (ReadableStream). Его можно прочитать
**один раз**. После вызова `.json()`, `.text()` или `.blob()` поток
"исчерпывается".

**Почему это важно для кэша:** если ты положишь в кэш `Response`,
а потом попытаешься прочитать тело — получишь ошибку. Кэшируй
**результат** парсинга:

```ts
const data = await response.json();  // прочитать один раз
cache.set(url, data);                // кэшировать данные, не Response
```

### 6.3. Сетевая ошибка vs HTTP-ошибка

```ts
// Сетевая ошибка — fetch rejected
fetch("https://несуществующий-домен.xyz")
  .catch(err => {
    // TypeError: Failed to fetch
    // err.name === "TypeError"
  });

// HTTP-ошибка — fetch fulfilled с плохим статусом
fetch("https://example.com/not-found")
  .then(response => {
    // response.status === 404
    // response.ok === false
    // НО промис fulfilled!
  });

// Отмена — fetch rejected с AbortError
const c = new AbortController();
c.abort();
fetch(url, { signal: c.signal })
  .catch(err => {
    // DOMException: The user aborted a request.
    // err.name === "AbortError"
  });
```

Три разных случая — три разных способа обработки. Твой код должен
различать все три.

## Глава 7. Паттерн: отменяемая пауза

### 7.1. Зачем это нужно

Между retry-попытками нужна пауза (backoff). Обычный `setTimeout`
нельзя отменить через AbortSignal. А отмена паузы необходима:

Сценарий:
1. Запрос к url1 получил 500 → ждём 2 секунды перед retry
2. В это время запрос к url2 получил 404 → нужно отменить ВСЁ
3. Но url1 "спит" 2 секунды и не знает, что уже пора остановиться

Без отменяемой паузы url1 проспит все 2 секунды, потом сделает retry
(который тоже будет отменён signal-ом), впустую потратив время.

### 7.2. Как сделать setTimeout отменяемым

Идея: обернуть `setTimeout` в промис и подписаться на `signal.abort`:

```
Создаём промис:
  ├── setTimeout(resolve, delay)     — "просыпайся через delay"
  └── signal.on("abort", reject)     — "или проснись раньше при отмене"

Кто первый сработает — тот и решает судьбу промиса.
```

Это **гонка** (race condition) между таймером и сигналом отмены.
Только один из двух callback-ов определит результат промиса (resolve
или reject). Промис "замерзает" после первого вызова resolve/reject —
второй вызов ничего не делает.

Но есть нюанс: если сработал таймер (resolve), нужно убрать listener
с signal (чтобы не было утечки памяти). И наоборот — если сработал
abort, нужно очистить setTimeout. Опция `{ once: true }` при добавлении
listener снимает его автоматически после первого вызова.

Подумай, как реализовать это, учитывая:
- Что если `signal.aborted` уже `true` на момент вызова?
- Нужно ли очищать `setTimeout` при abort?
- Что делать, если `signal` не передан (обычный sleep)?

## Глава 8. Архитектура решения

### 8.1. Разделение ответственности

Не пытайся написать одну большую функцию. Раздели на слои, где каждый
решает одну задачу:

```
Слой 3: fetchWithCache(urls)
  Ответственность:
  - Проверка кэша
  - Создание AbortController + таймаут
  - Запуск параллельных запросов
  - Сохранение результатов в кэш
  - Сбор и возврат результатов

Слой 2: fetchSingleUrl(url, signal, abortAll)
  Ответственность:
  - Один запрос с retry-логикой
  - Классификация ошибок (4xx vs 5xx vs AbortError)
  - Exponential backoff + jitter
  - При 4xx — сообщить "наверх" об отмене

Слой 1: fetch(url, { signal })
  Ответственность:
  - HTTP-запрос (уже реализован в браузере)

Вспомогательные:
  - sleep(ms, signal) — отменяемая пауза
  - LRUCache — кэш (уже реализован)
```

### 8.2. Поток данных

```
fetchWithCache(["url1", "url2", "url3"])
│
├─ url1 → кэш есть? ─── ДА ──→ вернуть из кэша
│                    └── НЕТ ──→ fetchSingleUrl(url1, signal)
│                                  ├── fetch → 200 → данные ✓
│                                  ├── fetch → 503 → sleep → retry
│                                  │                  └── fetch → 200 → данные ✓
│                                  ├── fetch → 404 → abort всех → ошибка ✗
│                                  └── AbortError → ошибка ✗
│
├─ url2 → (параллельно с url1)
│
└─ url3 → (параллельно с url1 и url2)
│
├── Promise.allSettled — ждём все
├── Успешные → сохранить в кэш
└── Вернуть результаты
```

### 8.3. Порядок реализации

Каждый шаг можно написать и протестировать отдельно:

**Шаг 1:** `sleep(ms)` — простая пауза

**Шаг 2:** `sleep(ms, signal)` — пауза с поддержкой отмены

**Шаг 3:** `fetchSingleUrl(url, signal)` — один запрос без retry.
Просто fetch + проверка статуса (2xx/4xx/5xx).

**Шаг 4:** Добавить retry в `fetchSingleUrl` — цикл попыток с backoff + jitter

**Шаг 5:** `fetchWithCache(urls)` — кэш + AbortController + таймаут +
Promise.allSettled + вызов fetchSingleUrl для каждого URL

### 8.4. Вопросы, на которые нужно ответить перед реализацией

Прежде чем писать код, ответь себе:

1. Что возвращает `fetchWithCache`? Массив результатов? Объект
   `{ url → data }`? Массив `{ status, value/reason }`?

2. Как `fetchSingleUrl` сообщает `fetchWithCache`, что нужно
   отменить все запросы при 4xx? (Подсказка: у них общий AbortController)

3. Что происходит, если кэш полон и приходят новые URL?
   (Ответ: LRU сам вытесняет старые — это его работа)

4. Нужно ли кэшировать ошибки? (Для минимальной реализации — нет.
   Но в продакшене иногда кэшируют 404, чтобы не долбить сервер
   повторными запросами к несуществующему ресурсу.)

5. В каком формате кэшировать? Строку (text)? Объект (json)?
   (Для простоты — json, если API отдаёт JSON.)

## Глава 9. URL для тестирования

Когда будешь тестировать — не нужно поднимать свой сервер.
Есть публичные сервисы:

**Всегда 200 + JSON:**
`https://jsonplaceholder.typicode.com/posts/1`

**Любой статус по запросу:**
`https://httpstat.us/500` — вернёт 500
`https://httpstat.us/404` — вернёт 404
`https://httpstat.us/503` — вернёт 503

**Искусственная задержка:**
`https://httpstat.us/200?sleep=15000` — ответит 200, но через 15 секунд
(хорошо для теста таймаута)
