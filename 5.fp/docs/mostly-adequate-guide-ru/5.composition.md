# [Глава 05: Использование композиции](https://github.com/MostlyAdequate/mostly-adequate-guide-ru/blob/master/ch05-ru.md)

Простой пример ф-ции `compose`

```js
const compose2 = (func1, func2) => (value) => func1(func2(value));
```

`func1, func2` - ф-ции, а `value` - значение, которое последовательно передается через каждую из них, как по "трубе".

Пример использования

```ts
const toUpperCase = (x: string) => x.toUpperCase();
const exclaim = (x: string) => `${x}!`;
const shout = compose(exclaim, toUpperCase);

shout("send in the clowns"); // "SEND IN THE CLOWNS!"
```

В нашем определении `compose`, ф-ция выполнится справа налево. Это читается куда проще, чем несколько вложенных вызовов.
Без композиции код бы выглядел так

```js
const shout = (x) => exclaim(toUpperCase(x));
```

Вместо того чтобы читать код изнутри наружу, мы читаем код справа налево.

**Композиция ассоциативна**

Допустим, у нас есть пример `1 + 2 + 3`

Мы можем посчитать это двумя способами:

1. Сначала сложить (1 + 2), получится 3, а потом прибивать 3. Результат 6.
2. Сначала сложить (2 + 3), а затем прибавить 1. Результат 6.

**Результат один и тот же**. Порядок действий не меняет результат.
Вот это свойство - "не важно, как расставить скобки" - и называется ассоциативностью.

## Теперь то же самое с функциями

**Порядок склеивания не важен, но порядок следования функция критически важен!**

`compose(A, B, C)` не равно`compose(B, A, C)`

Изменение порядка функций - это перестановка ступеней на заводском конвейере. Это уже другая логика и другой (или сломанный результат).

✅ КАК МОЖНО (Безопасное склеивание)

```js
const trim = (x) => x.trim();
const lower = (x) => x.toLowerCase();
const addExc = (x) => x + "!";

// Вариант 1: Сначала склеили 2 и 3, потом приклеили 1
const option1 = compose(trim, compose(lower, addExec));
const option2 = compose(compose(trim, lower), addExec);
const option3 = compose(trim, lower, addExc);

option1("  HELLO  "); // 'hello!'
option2("  HELLO  "); // 'hello!'
option3("  HELLO  "); // 'hello!'
```

❌ КАК НЕЛЬЗЯ (Опасная перестановка)

Менять ф-ции местами внутри списка

```js
const getLength = (x) => x.length; // Ожидает строку, возвращает число
const toLower = (x) => x.toLowerCase(); // Ожидает строку, возвращает строку

// ПРАВИЛЬНО: Строка -> становится маленькой -> считается длина
const correct = compose(getLength, toLower);
correct("HELLO"); // 5 (все ок)

// НЕПРАВИЛЬНО: Строка -> считается длина (получили число 5) -> пытаемся сделать число маленьким
const wrong = compose(toLower, getLength);
wrong("HELLO"); // 💥 ОШИБКА! toLowerCase не работает с числами
```

## Бесточечный стиль

Бесточечный стиль (он же pointfree — стиль отсутствия ссылок, комбинаторный стиль) подрузамевает ф-ции, которые никогда не упоминают данные, с которыми они работают.

```js
// не бесточечный стиль, поскольку мы упоминаем данные: word
const snakeCase = (word) => word.toLowerCase().replace(/\s+/gi, "_");

// бесточечный стиль
const snakeCase = compose(replace(/\s+/gi, "_"), toLowerCase);
```

Заметили, как мы частично применили `replace`?
Мы передаем данные от ф-ции к ф-ции, каждая из которых принимает только один аргумент.
Каррирование позволяет нам подготовить каждую ф-цию так, чтобы она ожидала только данные, производила с ними действия и передавала их дальше.
Еще стоит отметить что нам не нужны данные для построения нашей функции в бесточечной версии, тогда как в обычной версии нам требуется объявить`word`, прежде чем сделать хоть что-то.

Ещё один пример

```js
const initials = (name) => name.split(" ").map(compose(toUpperCase, head)).join(". ");

// Бесточечный стиль

const initials = compose(intercalate(". "), map(compose(toUpperCase, head)), split(" "));

initials("hunter stockton thompson"); // 'H. S. T'
```

Бесточечный стиль может помочь нам убрать излишнее именование значений, что сохранит краткость и обобщенность.
Это также индикатор функционального года, поскольку он делает очевидным, что некий конкретный фрагмент состоит только из небольших ф-ций,
которые принимают и возвращают данные.
К примеру, не получится просто так использовать цикл while в композиции. Однако имейте в виду, что бесточечный код — это обоюдоострый меч, и иногда он может сделать намерение одного разработчика неочевидным для других. Не весь функциональный код обязан быть бесточечным, и это нормально. Мы будем придерживаться этого стиля по мере возможности, а в остальных случаях будем использовать обычные функции.
