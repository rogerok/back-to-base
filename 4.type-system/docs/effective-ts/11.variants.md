# Вариантности

```ts
interface Entity {
  id: string;
}

interface User extends Entity {
  id: `u_${string}`;
  name: string;
}

interface CompanyUser extends User {
  company: string;
}

type Processor = (u: User) => User;

function processUser(u: Entity): CompanyUser {
  u.id = "8"; // Тут не возникает ошибка.

  return {
    id: "u_1",
    name: "Petya",
    company: "Dummy",
  };
}

const petya: User = { id: "u_1", name: "Petya" };

processUser(petya);
```

## Почему не возникает ошибка?

` u.id = '8';`
Из-за того, что объект передался по ссылке, мы потеряли данные.
Что на самом деле происходит?

Мы создаем пользователя с типом `User`, в памяти появился объект с id `u_1`, на него существует ссылка (`petya`). Когда
мы `petya` пеередаем в функцию `processUser`, в аргументе ф-ции `processUser` создается новая переменная `u` которая
содержит ссылку на объект `petya`,
но мы перезаписываем информацию о типе.
Из-за этого мы можем модифицировать тотже объект в памяти,

## Массивы ведут себя похожим образом

```ts
type Animal = { id: string; name: string };
type Cat = Animal & { name: `cat_${string}` };
type Dog = Animal & { name: `dog_${string}` };

declare let x: Cat[];

const p: Animal[] = x;
const newDog: Dog = { name: "dog_1", id: "123" };

p.push(newDog); // No Error
```

## In Out

TypeScript 4.7 ввёл явные аннотации вариантности через модификаторы `in` и `out` на параметрах обобщённых типов.

### `out T` — ковариантность (производитель)

`out` означает, что тип `T` используется **только в позиции вывода** (возвращаемое значение, readonly-поля).
Подтип `T` можно подставить вместо надтипа.

```ts
interface Producer<out T> {
  produce(): T;
}

const userProducer: Producer<User> = { produce: () => ({ id: "u_1", name: "Petya" }) };
const entityProducer: Producer<Entity> = userProducer; // OK — User extends Entity
```

### `in T` — контравариантность (потребитель)

`in` означает, что тип `T` используется **только в позиции ввода** (параметры функций).
Надтип `T` можно подставить вместо подтипа.

```ts
interface Consumer<in T> {
  consume(value: T): void;
}

const entityConsumer: Consumer<Entity> = { consume: (e) => console.log(e.id) };
const userConsumer: Consumer<User> = entityConsumer; // OK — Entity шире, чем User
```

### `in out T` — инвариантность

`in out` означает, что тип `T` используется **в обеих позициях**.
Подстановка допустима только для **точно совпадающего** типа.

```ts
interface Transformer<in out T> {
  transform(value: T): T;
}

const userTransformer: Transformer<User> = { transform: (u) => u };
// const entityTransformer: Transformer<Entity> = userTransformer; // Ошибка!
// const companyTransformer: Transformer<CompanyUser> = userTransformer; // Ошибка!
```

### Зачем это нужно

Без явных аннотаций TypeScript **выводит** вариантность сам, анализируя тело типа.
Явные модификаторы:

- делают намерение читаемым;
- ускоряют проверку типов (компилятор не анализирует тело);
- позволяют отловить ошибку, если реализация нарушает заявленную вариантность.

```ts
// Ошибка: out T, но T используется в позиции ввода
interface BrokenProducer<out T> {
  consume(value: T): void; // Error: Type 'T' is 'out' but used in 'in' position
}
```

### Связь с примером выше

В примере с `Processor = (u: User) => User`:

- параметр `u` — позиция `in` (контравариантна);
- возвращаемое значение — позиция `out` (ковариантна).

Поэтому `processUser: (u: Entity) => CompanyUser` совместим с `Processor`:

- `Entity` — надтип `User` (контравариантность параметра ✓);
- `CompanyUser` — подтип `User` (ковариантность возвращаемого значения ✓).
